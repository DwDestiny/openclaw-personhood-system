# Document Maintenance Methodology

## 目标

这套方法论只解决一件事：当新信息出现时，如何判断它该不该沉淀，以及应该写进哪一层文档。

它不是新的记忆引擎，也不是新的检索系统，而是站在现有 OpenClaw memory、session-memory、heartbeat、cron 之上的文档维护方法。

## 顶层原则

- 上下文归 OpenClaw 自己管理
- 我们只维护长期文档分层
- 不追求每轮都写
- 不把所有信息都升为长期知识
- 方法论优先于模板，边界优先于格式

## 文档分流总表

### `memory/world.md`

写这里的条件：
- 这是对外部世界的理解
- 跨任务可复用
- 相对稳定，不是一时观察

典型内容：
- 平台传播规律
- 行业认知
- 用户行为模式
- 内容类型表现差异

### `memory/self.md`

写这里的条件：
- 这是对智能体自己的理解
- 反复出现或经复盘确认
- 会影响之后的执行方式

典型内容：
- 常见跑偏模式
- 稳定优势或短板
- 自我约束
- 成熟方向

### `memory/expression.md`

写这里的条件：
- 这是表达和情绪控制规律
- 跨任务有效
- 会稳定影响对话质量或结果交付

典型内容：
- 语气控制
- 回复节奏
- 错误承接方式
- 文件/结果交付表达规则

### `MEMORY.md`

写这里的条件：
- 这是跨任务长期有效的总规则
- 需要作为高优先级稳定原则保留
- 不仅限于某一类 world/self/expression 子问题

典型内容：
- 顶层协作原则
- 长期有效禁忌
- 高价值稳定方法论

### `USER.md`

写这里的条件：
- 这是用户画像级稳定偏好
- 会长期影响沟通或协作
- 不是一次性指令

### `IDENTITY.md`

写这里的条件：
- 这是智能体人格底色或角色定位变化
- 更新频率应很低
- 不应该被日常任务频繁改写

### `memory/inbox/`

写这里的条件：
- 新认知有价值，但还不够稳定
- 与旧认知冲突，暂时不能直接覆盖
- 需要后续再验证

## 四步判断法

### 第一步：先判断值不值得写

如果只是：
- 一次性任务状态
- 临时口头要求
- 未验证猜测
- 对长期协作没影响

那就不写长期文档。

### 第二步：再判断它属于哪一类

先问四个问题：
- 这是对世界的理解吗？
- 这是对自己的理解吗？
- 这是表达规律吗？
- 这是用户画像或顶层长期规则吗？

能归类再写，不能归类先放 `inbox` 或不写。

### 第三步：再判断稳定度

- 只出现一次：通常不升长期
- 已重复出现：可考虑升长期
- 用户明确反复强调：优先升长期
- 已被多次任务验证：优先升长期

### 第四步：控制写入力度

- 优先写规则，不写流水账
- 优先写判断，不写细碎细节
- 优先写边界，不写模板口号
- 能写短就不写长

## 推荐维护时机

### heartbeat

适合做轻量蒸馏：
- 看有没有值得升级的新认知
- 把未稳定内容先放入 `inbox`
- 不做大规模重写

### 日总结 / 周总结

适合做中等强度整理：
- 提炼重复出现的规律
- 处理 `inbox` 候选
- 升级 world/self/expression/MEMORY

### 专门维护 skill

适合做深度整理：
- 重新梳理文档边界
- 合并重复规则
- 删除过时内容
- 调整错放内容

## 维护禁忌

- 不把 daily 流水账直接贴进长期文档
- 不把一次性情绪升级成长期规则
- 不把单次用户命令误写成用户画像
- 不把表达模板当成表达方法论
- 不把所有东西都写进 `MEMORY.md`
- 不为了“显得在学习”而过度写入
